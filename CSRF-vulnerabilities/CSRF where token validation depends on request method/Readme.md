# Lab Write-Up: CSRF where token validation depends on request method

**Platform:** PortSwigger Web Security Academy
**Vulnerability:** Cross-Site Request Forgery (CSRF)
**Difficulty:** Apprentice

---

## 1. Executive Summary / Objective

* **Goal:** Extension of CSRF-vulnerability-with no defense lab , where goal is to bypass the Anti-CSRF token by changing  `POST` method to a `GET` method.
* **Vulnerability Type:** [OWASP A01:2021 – Broken Access Control](https://owasp.org/Top10/A01_2021-Broken_Access_Control/)/Cross-Site Request Forgery.
* **Root Cause:** The application implements Anti-CSRF token verification system but restricts the token validation upon certain request method(POST in this case).If the endpoint encounters **a `GET` request, it skips token validation entirely.**

---

## 2. Analysis & Discovery

### Spotting the Attack Surface

Similar to the lab CSRF-vulnerability with no defense, I analyzed state-changing actions across the user profile interface (such as posting comments, User login, and updating profile information).The email update mechanism emerged as a primary target. The request generated by it looked CSRF-proof initially due to the presence of an Anti-CSRF token but still that might be vulnerable.

### Reading the Clues

1. **[State-Changing action]:** The `/my-account/change-email` endpoint handles a permanent, unauthorized-sensitive change to user data.
2. **[Session + Token based Authentication]:** The application relies entirely on the `session` cookie and anti-CSRF token to authenticate the identity of the user initiating the request.
3. **[Lack of SameSite attribute]:** No modern security measures , like `SameSite` flags (`Strict` or `Lax`) for session cookies,meaning the browser can attach cookies to the cross-site requests initiated by third-party domains.
4. **[Method Flaw ]:** Sending this request to Burp Repeater and changing the method from `POST` to `GET` (while removing the `csrf` parameter entirely) results in a `200 OK` or a successful `302 Redirect`. This confirms that the backend application fails to validate the token when an alternative HTTP method is used.

## 3. Exploitation

### Step 1 — Moving to Exploit Server

The lab provided an exploit server that serves as the *"attacker's malicious website"* I navigated to it and opened the HTML body editor.

### Step 2 — Crafting the Exploit Payload

My goal was to create a HTML page that will force the victim's browser to send a  GET request to the target site with my pre-filled payload.

```html
<form action="https://0a6c00b1043fc62c8207b69c005a00a7.web-security-academy.net/my-account/change-email" method = "GET">
    <input type="hidden" name="email" value="attacked@gmail.com">
</form>
<script>
        document.forms[0].submit();
</script>
```

**Payload Breakdown:**

* **[<form>]:** The target URL points to the vulnerable application's endpoint.The `method = "GET"` is declared to prevent the browser from making `POST` request.
* **[input type = hidden]:** prefills the email field with my value and the victim never sees it.
* **[Auto-submit Script]:** The `<script>` block calls `document.forms[0].submit()` immediately on page load — removing any requirement for manual user interaction.

### Step 3 — Deliver and Verify

I saved the exploit on the exploit server and used the **"Deliver to victim"** option to simulate the victim visiting the page. The server processed the forged request and updated the victim's email — the lab was solved.

## 4. Remediation

To secure the application against this vulnerability, developers could implement structural changes that enforce strict method restriction and decouple state-changing validation logic from the client-selected HTTP method.

### Fix 1 — Strict Anti-CSRF Token Validation & Method Enforcement (Primary Defense)

The application must reject any state-changing request submitted via `GET`. The anti-CSRF token verification must apply universally across all valid state-changing routes, regardless of the method parameter received.

```javascript
// Middleware to enforce strict CSRF checking across routes
app.post('/my-account/change-email', ensureAuthenticated, verifyCsrfToken, (req, res) => {
    const { email } = req.body;
    // Process secure email change execution
    db.updateUserEmail(req.user.id, email);
    res.status(200).send("Email updated successfully.");
});

// Explicitly block GET requests from processing state modifications
app.get('/my-account/change-email', (req, res) => {
    res.status(405).send("Method Not Allowed. State changes are prohibited on GET requests.");
});
```

### Fix 2 — SameSite cookie Attributes

To provide protection even if application-layer validation fails, session cookies must be hardened with contemporary browser security flags. Setting the `SameSite` attribute prevents the browser from automatically sending authentication context during unauthorized cross-site transitions.

```http
Set-Cookie: session=abc123; SameSite=Strict;Secure;
```

**In short:** Relying on security defenses that vary based on client-controlled parameters can introduce dangerous structural gaps; applications must also enforce universal token validation to remain secure.
