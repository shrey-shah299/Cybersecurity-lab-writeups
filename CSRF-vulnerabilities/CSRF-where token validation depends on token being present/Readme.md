# Lab Write-Up: CSRF where token validation depends on token being present
**Platform:** PortSwigger Web Security Academy  
**Vulnerability:** Cross-Site Request Forgery (CSRF)  
**Difficulty:** PRACTITIONER

---

## 1. Executive Summary / Objective

* **Goal:** Extension of CSRF-vulnerability-with no defense lab , here the goal is to find the flaw - where CSRF token validation depends on the presence of tokens.
* **Vulnerability Type:** OWASP A01:2021 – Broken Access Control/Cross-Site Request Forgery.
* **Root Cause:** This site provides a account email update feature , which is vulnerable to the CSRF attack even though it contains the anti-CSRF token. The flaw is the token is only validated when the token is present else the validation is bypassed.

---

## 2. Analysis & Discovery

### Spotting the Attack Surface
Similar to the lab CSRF-vulnerability with no defense, I analyzed state-changing actions across the user profile interface (such as posting comments, User login, and updating profile information).The email update mechanism emerged as a primary target. The request generated by it looked CSRF-proof initially due to the presence of an Anti-CSRF token but further testing revealed a flaw.

### Intercepting the Request
The request intercepted for the update-email button contained the following structure:

```http

Host: 0a5f00db04f11d11806330c900a000da.web-security-academy.net

Sec-Ch-Ua-Mobile: ?0
Sec-Ch-Ua-Platform: "Windows"

Content-Type: application/x-www-form-urlencoded
Upgrade-Insecure-Requests: 1
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/146.0.0.0 Safari/537.36
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,image/avif,image/webp,image/apng,*/*;q=0.8,application/signed-exchange;v=b3;q=0.7

email=hello%40gmail.com&csrf=DbPa2M8A6nr6a6Pj6JrmQsas7oiz83wa
```

### Reading the Clues

1. **[State-Changing action]:** The `/my-account/change-email` endpoint handles a permanent, unauthorized, sensitive change to user data.
2. **[Session + Token based Authentication]:** The application relies entirely on the `session` cookie and anti-CSRF token to authenticate the identity of the user initiating the request.
3. **[Lack of SameSite attribute]:** No modern security measures , like `SameSite` flags (`Strict` or `Lax`) for session cookies,meaning the browser can attach cookies to the cross-site requests initiated by third-party domains.
4. **[Method Flaw]:** Sending this request to Burp Repeater and removing the `csrf` parameter resulted in a successful `302 Redirect`. This confirms that the backend application fails to validate request when no CSRF token is present - Vulnerability found.

## 3. Exploitation

### Step 1 — Moving to Exploit Server

The lab also provided an exploit server that serves as the *"attacker's malicious website"* I navigated to it and opened the HTML body editor.

### Step 2 — Crafting the Exploit Payload
As we got to know that the token validation entirely depends on the presence of the CSRF token . This validation can be totally bypassed by not providing any CSRF token , so the payload here was similar to the one provided in the lab - CSRF with no defense.

```html
<form action="https://0a5f00db04f11d11806330c900a000da.web-security-academy.net/my-account/change-email" method = "POST">
    <input type="hidden" name="email" value="attacked@gmail.com">
</form>
<script>
        document.forms[0].submit();
</script>
```

**Payload Breakdown:**

* **[Form tag]:** The target URL points to the vulnerable application's endpoint.The `method = "POST"` is declared to force the browser to make a "POST" request.
* **[input type = hidden]:** prefills the email field with my value and the victim never sees it.
* **[Auto-submit Script]:** The `<script>` block calls `document.forms[0].submit()` immediately on page load — removing any requirement for manual user interaction.


### Step 3 — Deliver and Verify

I saved the exploit on the exploit server and used the **"Deliver to victim"** option to simulate the victim visiting the page. The server processed the forged request and updated the victim's email — the lab was solved.

## 4. Remediation



### Fix 1 — Strict Server Side Token Validation
The application must enforce a "fail-secure" logic: validation should not be skipped if the token is omitted. Instead, the backend must verify that the token is present, in the correct format, and cryptographically matches the token assigned to the user's current session. If the token is missing or invalid, the server must reject the request with an HTTP 403 Forbidden status.
```javascript
function strictCsrfValidation(req, res, next) {
    const submittedToken = req.body.csrf;
    const sessionToken = req.session.csrfToken;

    // Fail-secure: Reject if token is missing entirely OR if it does not match
    if (!submittedToken || submittedToken !== sessionToken) {
        return res.status(403).send('Forbidden: Invalid or missing CSRF token');
    }
    
    // Proceed if the token is present and valid
    next();
}
```

### Fix 2 — SameSite cookie Attributes

To provide protection even if application-layer validation fails, session cookies must be hardened with contemporary browser security flags. Setting the `SameSite` attribute prevents the browser from automatically sending authentication context during unauthorized cross-site transitions.

```http
Set-Cookie: session=abc123; SameSite=Strict;Secure;
```


**In short:** Relying on security defenses that vary based on client-controlled parameters can introduce dangerous structural gaps; applications must not only enforce universal token validation to remain secure but also strict checks for the presence of tokens.
