# Lab Write-Up: CSRF-where token is not tied to user session

**Platform:** PortSwigger Web Security Academy  
**Vulnerability:** Cross-Site Request Forgery (CSRF)  
**Difficulty:** PRACTITIONER

---

## 1. Executive Summary / Objective

* **Goal:** Extension of CSRF-vulnerability-with no defense lab , here the objective is to find the flaw - where CSRF token validation is not tied to the user session
* **Vulnerability Type:** OWASP A01:2021 – Broken Access Control/Cross-Site Request Forgery.
* **Root Cause:** This site provides an account email update feature with anti-CSRF tokens. However, a critical logical flaw exists: the token is not  tied to the user's session. Instead, the backend simply validates the token against a global pool of unused tokens stored in its database.

---

## 2. Analysis & Discovery

### Spotting the Attack Surface
Similar to previous CSRF labs, I analyzed state-changing actions across the user profile interface (such as posting comments, User login, and updating profile information).The email update mechanism emerged as a primary target. The request generated by it looked CSRF-proof initially due to the presence of an Anti-CSRF token,but further testing revealed a flaw.

### Reading the Clues
1. **[State-Changing action]:** The `/my-account/change-email` endpoint handles a permanent, unauthorized, sensitive change to user data.
2. **[Session + Token based Authentication]:** The application relies entirely on the `session` cookie and anti-CSRF token to authenticate the identity of the user initiating the request.
3. **[The Single-Use Token illusion]:** These CSRF tokens were strictly single-use. If the token was once used by the token then it will be immediately invalidated by the server.
4. **[Lack of SameSite attribute]:** No modern security measures , like `SameSite` flags (`Strict` or `Lax`) for session cookies,meaning the browser can attach cookies to the cross-site requests initiated by third-party domains.
5. **[Method Flaw]:** To test the session-binding theory . I intercepted the email change request from a different user, copied the fresh CSRF token and dropped the request so that the token remains unused. I pasted this token in the victim's intercepted request . The server responded with a `302 Found Redirect`  .

## 3. Exploitation

### Step 1 — Moving to the Exploit Server
The lab also provided an exploit server that serves as the *"attacker's malicious website"* I navigated to it and opened the HTML body editor.

### Step 2 — Crafting the Exploit Payload
As I got to know that the token validation entirely depends on the token being present on the database . This can be exploited by using a fresh token from a different user.I intercepted a request from a different account(attacker's account) for the same functionality and used that token in my exploit code. 

```html
<form action="https://0a9300fd032ed63b800d122e00b80070.web-security-academy.net/my-account/change-email" method = "POST">
    <input type="hidden" name="email" value="attacked@gmail.com">
    <input type="hidden" name="csrf" value="ncW9PMDd38RMdsgOIgrThXhsCPNFBFTg">
</form>
<script>
        document.forms[0].submit();
</script>
```

**Payload Breakdown:**
* **[Form tag]:** The target URL points to the vulnerable application's endpoint.The `method = "POST"` is declared to force the browser to make a "POST" request.
* **[input type = hidden - csrf]:** prefills the csrf token field with my value(the csrf token taken from the attacker's own website) and the victim never sees it.
* **[Auto-submit Script]:** The `<script>` block calls `document.forms[0].submit()` immediately on page load — removing any requirement for manual user interaction.

### Step 3 — Deliver and Verify
I saved the exploit on the exploit server and used the **"Deliver to victim"** option to simulate the victim visiting the page. The server processed the forged request and updated the victim's email — the lab was solved.

## 4. Remediation



### Fix 1 — Primary Defense Strategy
The vulnerability exists because tokens are validated globally rather than individually. The primary fix is to logically bind the anti-CSRF token to the specific user's session. The server must not only validate the correctness of the token but also verify that it belongs to the specific user's active session.
```javascript
// Generate token and store it in the user's specific session object
req.session.csrfToken = generateCryptographicToken();

//  Validation logic on state-changing endpoints
function strictCsrfValidation(req, res, next) {
    const submittedToken = req.body.csrf;
    const sessionToken = req.session.csrfToken;

    // Reject if token is missing OR if it doesn't match the specific user's session
    if (!submittedToken || submittedToken !== sessionToken) {
        return res.status(403).send('Forbidden: Invalid CSRF token for this session');
    }
    
    next();
}
```

### Fix 2 — SameSite Cookie Attributes

To provide protection even if application-layer validation fails, session cookies must be hardened with contemporary browser security flags. Setting the `SameSite` attribute prevents the browser from automatically sending authentication context during unauthorized cross-site transitions.

```http
Set-Cookie: session=abc123; SameSite=Strict;Secure;
```

**In short:** This flaw in the Anti-CSRF token validation method is because the tokens are not tied to the user session but are validated from a pool of global tokens.
