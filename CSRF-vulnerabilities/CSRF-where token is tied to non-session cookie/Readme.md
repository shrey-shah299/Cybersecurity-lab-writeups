# Lab Write-Up: CSRF where token is tied to non-session cookie

**Platform:** PortSwigger Web Security Academy  
**Vulnerability:** Cross-Site Request Forgery (CSRF)  
**Difficulty:** PRACTITIONER
---

## 1. Executive Summary / Objective

* **Goal:** Extension of the CSRF vulnerability with no defense lab. Here, the objective is to find the flaw where the CSRF token is tied to the user's non-session cookie.
* **Vulnerability Type:** OWASP A01:2021 – Broken Access Control / Cross-Site Request Forgery.
* **Root Cause:** This site provides an account email update feature with anti-CSRF tokens. However, a critical logical flaw exists: even though the token is tied to a cookie, it is tied to a non-session cookie. This can occur when the application employs two different frameworks — one for session management and another for CSRF protection.

---

## 2. Analysis & Discovery

### Spotting the Attack Surface
Similar to previous CSRF labs, I analyzed state-changing actions across the user profile interface (such as posting comments, user login, updating profile information, and search functionality). The email update mechanism emerged as a primary target. The request generated by it looked CSRF-proof initially due to the presence of an anti-CSRF token, but further testing revealed a flaw.
The search menu emerged as a secondary target. From the interception of its request, it was found that this functionality induced a `Set-Cookie` header to set a cookie for the last-searched item, thus emerging as a surface for potential injection.

### Reading the Clues
1. **[State-Changing Action]:** The `/my-account/change-email` endpoint handles a permanent, unauthorized, sensitive change to user data.
2. **[Session + Token-Based Authentication]:** The application relies entirely on the `session` cookie and anti-CSRF token to authenticate the identity of the user initiating the request.
3. **[Lack of SameSite Attribute]:** No modern security measures, like `SameSite` flags (`Strict` or `Lax`) for session cookies, meaning the browser can attach cookies to cross-site requests initiated by third-party domains.
4. **[Method Flaw]:** To test the session-binding theory, I intercepted the email change request from a different user and copied the fresh `csrfKey` and token. I then pasted the attacker's `csrfKey` and token into the victim's intercepted request. The server responded with a `302 Found` redirect, clearly indicating the lack of a single unified framework for both session management and CSRF validation.
5. **[Search Functionality]:** I intercepted the search menu request and found it induces a `Set-Cookie` header to set a cookie for `last-searched-item`, thus emerging as a surface for an attacker to inject a cookie into the victim's browser.

## 3. Exploitation

### Step 1 — Moving to the Exploit Server and Initial Testing
The lab also provided an exploit server that serves as the *"attacker's malicious website"*. I navigated to it and opened the HTML body editor.

I checked the intercepted request for email-update for the following vulnerabilities:
1. Check if token absence can provide a bypass — but it failed since it returned a `403 Forbidden` response.
2. Check if changing the request method works — but it failed since it returned a `405 Method Not Allowed` response.
3. Check if the token is tied to a non-session cookie — this was **successful**, as when the non-session cookie `csrfKey` and CSRF token from another user account were used in this request, it returned `302 Found`.

### Step 2 — Finding a Vulnerable Endpoint for Cookie Injection

I navigated to the other features on the home page. The search functionality was found to be one.
I updated the intercepted request to check if it allows for the setting of cookies:

```http
GET /?search=hat%0d%0aSet-Cookie:%20csrfKey=SQm00rmRtPeICyLnTlJYJfGSrdmoUbft HTTP/2
```
This endpoint would allow me to set a cookie in the victim's browser.

### Step 3 — Crafting the Exploit Payload
Since the email update functionality was CSRF-vulnerable (the CSRF token was tied to a non-session cookie which can be injected into the victim's browser using the search URL) and the session cookie had no `SameSite` flag (meaning it can be sent with cross-site requests), I crafted the following exploit payload:

```html
<form action="https://0ad100be03b0fe4580a626a200b0006a.web-security-academy.net/my-account/change-email" method="POST" id="csrf-form">
    <input type="hidden" name="email" value="attacked@gmail.com">
    <input type="hidden" name="csrf" value="bTJHQufl4fcp1yGjXrRYkmuJhFKszCcp">
</form>
<img style="display:none;" src="https://0ad100be03b0fe4580a626a200b0006a.web-security-academy.net/?search=hat%0d%0aSet-Cookie:%20csrfKey=9Qm00rmRtPeICyLnT1JYJfG9rdmoUbft%3b%20SameSite=None" onerror="document.forms[0].submit()">
```

**Payload Breakdown:**
* **[Form tag]:** The target URL points to the vulnerable application's endpoint. The `method="POST"` is declared to force the browser to make a POST request.
* **[input type="hidden" - csrf]:** Pre-fills the CSRF token field with the attacker's value (the CSRF token taken from the attacker's own account); the victim never sees it.
* **[img tag]:** The image tag is used to make a `GET` request to the vulnerable search endpoint for cookie injection. The `Set-Cookie` header contains the `csrfKey` taken from the attacker's browser, and the `onerror` handler auto-submits the form once the cookie has been set.

### Step 4 — Deliver and Verify
I saved the exploit on the exploit server and used the **"Deliver to victim"** option to simulate the victim visiting the page. The server processed the forged request and updated the victim's email — the lab was solved.

## 4. Remediation

The root cause of this vulnerability is the use of two independent frameworks — one for session management and another for CSRF protection — without binding the CSRF token to the user's authenticated session. A robust remediation strategy must eliminate this decoupling entirely.

### Fix 1 — Primary Defense: Bind the CSRF Token Strictly to the Session

The CSRF token **must** be cryptographically tied to the user's server-side session, not an independent, injectable cookie. On every state-changing request, the server must verify that the submitted token matches the one stored server-side for that specific session. This means an attacker who can inject their own `csrfKey` cookie into the victim's browser gains nothing, because the token they supply will never match the victim's session.

**Implementation approach (Node.js / Express example using `express-session` + custom token):**

```javascript
const crypto = require('crypto');
const session = require('express-session');

// Middleware: generate a CSRF token and bind it to the session
function generateCsrfToken(req, res, next) {
  // Only generate a new token if one doesn't already exist for this session
  if (!req.session.csrfToken) {
    req.session.csrfToken = crypto.randomBytes(32).toString('hex');
  }
  // Expose the token to views via a local variable
  res.locals.csrfToken = req.session.csrfToken;
  next();
}

// Middleware: validate the CSRF token on every mutating request
function validateCsrfToken(req, res, next) {
  const methods = ['POST', 'PUT', 'PATCH', 'DELETE'];
  if (!methods.includes(req.method)) return next();

  const submittedToken = req.body.csrf || req.headers['x-csrf-token'];
  const sessionToken   = req.session.csrfToken;

  // Use timing-safe comparison to prevent timing attacks
  if (
    !submittedToken ||
    !sessionToken   ||
    !crypto.timingSafeEqual(
      Buffer.from(submittedToken),
      Buffer.from(sessionToken)
    )
  ) {
    return res.status(403).json({ error: 'Invalid or missing CSRF token.' });
  }

  next();
}

// Apply globally
app.use(session({ secret: process.env.SESSION_SECRET, resave: false, saveUninitialized: false }));
app.use(generateCsrfToken);
app.use(validateCsrfToken);
```

**Key properties of this approach:**
| Property | Why it matters |
|---|---|
| Token stored **server-side in the session** | An attacker cannot read or predict it, even if they can set cookies |
| Token is **session-scoped** | Injecting a foreign `csrfKey` cookie is useless — the server validates against *the victim's session* |
| `timingSafeEqual` comparison | Prevents timing-based token oracle attacks(using time required to predict the token) |
| Token rotated on each session | Old tokens become invalid after logout or session expiry |

> **In short:** Never store the CSRF token's ground-truth in a client-controlled cookie. Tie it exclusively to the server-side session so that no amount of cookie injection can forge a valid token pair.

### Fix 2 — SameSite Cookie Attributes
To provide protection even if application-layer validation fails, session cookies must be hardened with modern browser security flags. Setting the `SameSite` attribute prevents the browser from automatically sending authentication context during unauthorized cross-site transitions.

```http
Set-Cookie: session=abc123; SameSite=Strict; Secure; HttpOnly
```

**In short:** Defense-in-depth is essential. Binding the CSRF token to the session eliminates the root cause, while `SameSite=Strict` and `HttpOnly` flags act as an additional safety net — ensuring that even if one layer is misconfigured, the attack chain cannot be completed.
